Features can also be included individually to shorten compilation times.

 \hypertarget{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md75}{}\doxysubsection{$<$a name=\char`\"{}section2\char`\"{}$>$$<$/a$>$ 2. Swizzling}\label{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md75}
Shader languages like GLSL often feature so-\/called swizzle expressions, which may be used to freely select and arrange a vector\textquotesingle{}s components. For example, {\ttfamily variable.\+x}, {\ttfamily variable.\+xzy} and {\ttfamily variable.\+zxyy} respectively form a scalar, a 3D vector and a 4D vector. The result of a swizzle expression in GLSL can be either an R-\/value or an L-\/value. Swizzle expressions can be written with characters from exactly one of {\ttfamily xyzw} (usually for positions), {\ttfamily rgba} (usually for colors), and {\ttfamily stpq} (usually for texture coordinates).

To prevent name collisions and ensure a consistent API, the name not\+\_\+ (note the underscore) is used instead.\hypertarget{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md118}{}\doxysubsubsection{$<$a name=\char`\"{}section6\+\_\+2\char`\"{}$>$$<$/a$>$ 6.\+2. Precision qualifiers support}\label{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md118}
GLM supports GLSL precision qualifiers through prefixes instead of qualifiers. For example, GLM exposes \textbackslash{}verb$\vert$lowp\+\_\+vec4$\vert$, \textbackslash{}verb$\vert$mediump\+\_\+vec4$\vert$ and \textbackslash{}verb$\vert$highp\+\_\+vec4$\vert$ as variations of \textbackslash{}verb$\vert$vec4$\vert$.

Similarly to GLSL, GLM precision qualifiers are used to trade precision of operations in term of \href{http://en.wikipedia.org/wiki/Unit_in_the_last_place}{\texttt{ ULPs}} for better performance. By default, all the types use high precision.

Following GLSL conventions is a really strict policy of GLM. It has been designed following the idea that everyone does its own math library with his own conventions. The idea is that brilliant developers (the Open\+GL ARB) worked together and agreed to make GLSL. Following GLSL conventions is a way to find consensus. Moreover, basically when a developer knows GLSL, he knows GLM.\hypertarget{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md122}{}\doxysubsubsection{$<$a name=\char`\"{}section7\+\_\+2\char`\"{}$>$$<$/a$>$ 7.\+2. Does GLM run GLSL program?}\label{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md122}
No, GLM is a C++ implementation of a subset of GLSL.\hypertarget{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md123}{}\doxysubsubsection{$<$a name=\char`\"{}section7\+\_\+3\char`\"{}$>$$<$/a$>$ 7.\+3. Does a GLSL compiler build GLM codes?}\label{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md123}
No, this is not what GLM attends to do.\hypertarget{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md124}{}\doxysubsubsection{$<$a name=\char`\"{}section7\+\_\+4\char`\"{}$>$$<$/a$>$ 7.\+4. Should I use ‘\+GTX’ extensions?}\label{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md124}
GTX extensions are qualified to be experimental extensions. In GLM this means that these extensions might change from version to version without any restriction. In practice, it doesn’t really change except time to time. GTC extensions are stabled, tested and perfectly reliable in time. Many GTX extensions extend GTC extensions and provide a way to explore features and implementations and APIs and then are promoted to GTC extensions. This is fairly the way Open\+GL features are developed; through extensions.

Stating with GLM 0.\+9.\+9, to use experimental extensions, an application must define GLM\+\_\+\+ENABLE\+\_\+\+EXPERIMENTAL.\hypertarget{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md125}{}\doxysubsubsection{$<$a name=\char`\"{}section7\+\_\+5\char`\"{}$>$$<$/a$>$ 7.\+5. Where can I ask my questions?}\label{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md125}
A good place is \href{http://stackoverflow.com/search?q=GLM}{\texttt{ stackoverflow}} using the GLM tag.\hypertarget{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md126}{}\doxysubsubsection{$<$a name=\char`\"{}section7\+\_\+6\char`\"{}$>$$<$/a$>$ 7.\+6. Where can I find the documentation of extensions?}\label{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md126}
The Doxygen generated documentation includes a complete list of all extensions available. Explore this \href{http://glm.g-truc.net/html/index.html}{\texttt{ {\itshape API documentation}}} to get a complete view of all GLM capabilities!\hypertarget{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md127}{}\doxysubsubsection{$<$a name=\char`\"{}section7\+\_\+7\char`\"{}$>$$<$/a$>$ 7.\+7. Should I use ‘using namespace glm;’?}\label{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md127}
NO! Chances are that if using namespace glm; is called, especially in a header file, name collisions will happen as GLM is based on GLSL which uses common tokens for types and functions. Avoiding using namespace glm; will a higher compatibility with third party library and SDKs.\hypertarget{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md128}{}\doxysubsubsection{$<$a name=\char`\"{}section7\+\_\+8\char`\"{}$>$$<$/a$>$ 7.\+8. Is GLM fast?}\label{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md128}
GLM is mainly designed to be convenient and that\textquotesingle{}s why it is written against the GLSL specification.

Following the Pareto principle where 20\% of the code consumes 80\% of the execution time, GLM operates perfectly on the 80\% of the code that consumes 20\% of the performances. Furthermore, thanks to the lowp, mediump and highp qualifiers, GLM provides approximations which trade precision for performance. Finally, GLM can automatically produce SIMD optimized code for functions of its implementation.

However, on performance critical code paths, we should expect that dedicated algorithms should be written to reach peak performance.\hypertarget{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md129}{}\doxysubsubsection{$<$a name=\char`\"{}section7\+\_\+9\char`\"{}$>$$<$/a$>$ 7.\+9. When I build with Visual C++ with /\+W4 warning level, I have warnings...}\label{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md129}
You should not have any warnings even in /\+W4 mode. However, if you expect such level for your code, then you should ask for the same level to the compiler by at least disabling the Visual C++ language extensions (/\+Za) which generates warnings when used. If these extensions are enabled, then GLM will take advantage of them and the compiler will generate warnings.\hypertarget{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md130}{}\doxysubsubsection{$<$a name=\char`\"{}section7\+\_\+10\char`\"{}$>$$<$/a$>$ 7.\+10. Why some GLM functions can crash because of division by zero?}\label{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md130}
GLM functions crashing is the result of a domain error. Such behavior follows the precedent set by C and C++\textquotesingle{}s standard library. For example, it’s a domain error to pass a null vector (all zeroes) to glm\+::normalize function, or to pass a negative number into std\+::sqrt.\hypertarget{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md131}{}\doxysubsubsection{$<$a name=\char`\"{}section7\+\_\+11\char`\"{}$>$$<$/a$>$ 7.\+11. What unit for angles is used in GLM?}\label{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md131}
GLSL is using radians but GLU is using degrees to express angles. This has caused GLM to use inconsistent units for angles. Starting with GLM 0.\+9.\+6, all GLM functions are using radians. For more information, follow the \href{http://www.g-truc.net/post-0693.html\#menu}{\texttt{ link}}.\hypertarget{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md132}{}\doxysubsubsection{$<$a name=\char`\"{}section7\+\_\+12\char`\"{}$>$$<$/a$>$ 7.\+12. Windows headers cause build errors...}\label{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md132}
Some Windows headers define min and max as macros which may cause compatibility with third party libraries such as GLM. It is highly recommended to \href{http://stackoverflow.com/questions/4913922/possible-problems-with-nominmax-on-visual-c}{\texttt{ define NOMINMAX}} before including Windows headers to workaround this issue. To workaround the incompatibility with these macros, GLM will systematically undef these macros if they are defined.\hypertarget{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md133}{}\doxysubsubsection{$<$a name=\char`\"{}section7\+\_\+13\char`\"{}$>$$<$/a$>$ 7.\+13. Constant expressions support}\label{md___users_cwb_developer_learning_graphic_render_engine_ceres_3rdparty_glm_manual_autotoc_md133}
GLM has some C++ \href{http://en.cppreference.com/w/cpp/language/constexpr}{\texttt{ constant expressions}} support. However, GLM automatically detects the use of SIMD instruction sets through compiler arguments to populate its implementation with SIMD intrinsics. Unfortunately, GCC and Clang doesn\textquotesingle{}t support SIMD instrinsics as constant expressions. To allow constant expressions on all vectors and matrices types, define GLM\+\_\+\+FORCE\+\_\+\+PURE before including GLM headers.
