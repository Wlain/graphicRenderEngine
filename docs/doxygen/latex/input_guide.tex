GLFW provides many kinds of input. While some can only be polled, like time, or only received via callbacks, like scrolling, many provide both callbacks and polling. Callbacks are more work to use than polling but is less CPU intensive and guarantees that you do not miss state changes.

All input callbacks receive a window handle. By using the \mbox{\hyperlink{window_guide_window_userptr}{window user pointer}}, you can access non-\/global structures or objects from your callbacks.

To get a better feel for how the various events callbacks behave, run the {\ttfamily events} test program. It register every callback supported by GLFW and prints out all arguments provided for every event, along with time and sequence information.\hypertarget{input_guide_events}{}\doxysection{Event processing}\label{input_guide_events}
GLFW needs to poll the window system for events both to provide input to the application and to prove to the window system that the application hasn\textquotesingle{}t locked up. Event processing is normally done each frame after \mbox{\hyperlink{window_guide_buffer_swap}{buffer swapping}}. Even when you have no windows, event polling needs to be done in order to receive monitor and joystick connection events.

There are three functions for processing pending events. \mbox{\hyperlink{group__window_ga872d16e4c77f58c0436f356255920cfc}{glfw\+Poll\+Events}}, processes only those events that have already been received and then returns immediately.

This function supports everything from single lines up to and including the unmodified contents of the whole {\ttfamily gamecontrollerdb.\+txt} file.

Below is a description of the mapping format. Please keep in mind that {\bfseries{this description is not authoritative}}. The format is defined by the SDL and SDL\+\_\+\+Game\+Controller\+DB projects and their documentation and code takes precedence.

Each mapping is a single line of comma-\/separated values describing the GUID, name and layout of the gamepad. Lines that do not begin with a hexadecimal digit are ignored.

The first value is always the gamepad GUID, a 32 character long hexadecimal string that typically identifies its make, model, revision and the type of connection to the computer. When this information is not available, the GUID is generated using the gamepad name. GLFW uses the SDL 2.\+0.\+5+ GUID format but can convert from the older formats.

The second value is always the human-\/readable name of the gamepad.

All subsequent values are in the form {\ttfamily \texorpdfstring{$<$}{<}field\texorpdfstring{$>$}{>}\+:\texorpdfstring{$<$}{<}value\texorpdfstring{$>$}{>}} and describe the layout of the mapping. These fields may not all be present and may occur in any order.

The button fields are {\ttfamily a}, {\ttfamily b}, {\ttfamily c}, {\ttfamily d}, {\ttfamily back}, {\ttfamily start}, {\ttfamily guide}, {\ttfamily dpup}, {\ttfamily dpright}, {\ttfamily dpdown}, {\ttfamily dpleft}, {\ttfamily leftshoulder}, {\ttfamily rightshoulder}, {\ttfamily leftstick} and {\ttfamily rightstick}.

The axis fields are {\ttfamily leftx}, {\ttfamily lefty}, {\ttfamily rightx}, {\ttfamily righty}, {\ttfamily lefttrigger} and {\ttfamily righttrigger}.

The value of an axis or button field can be a joystick button, a joystick axis, a hat bitmask or empty. Joystick buttons are specified as {\ttfamily bN}, for example {\ttfamily b2} for the third button. Joystick axes are specified as {\ttfamily aN}, for example {\ttfamily a7} for the eighth button. Joystick hat bit masks are specified as {\ttfamily h\+N.\+N}, for example {\ttfamily h0.\+8} for left on the first hat. More than one bit may be set in the mask.

Before an axis there may be a {\ttfamily +} or {\ttfamily -\/} range modifier, for example {\ttfamily +a3} for the positive half of the fourth axis. This restricts input to only the positive or negative halves of the joystick axis. After an axis or half-\/axis there may be the {\ttfamily $\sim$} inversion modifier, for example {\ttfamily a2$\sim$} or {\ttfamily -\/a7$\sim$}. This negates the values of the gamepad axis.

The hat bit mask match the \mbox{\hyperlink{group__hat__state}{hat states}} in the joystick functions.

There is also the special {\ttfamily platform} field that specifies which platform the mapping is valid for. Possible values are {\ttfamily Windows}, {\ttfamily Mac OS X} and {\ttfamily Linux}.

\hypertarget{input_guide_time}{}\doxysection{Time input}\label{input_guide_time}
GLFW provides high-\/resolution time input, in seconds, with \mbox{\hyperlink{group__input_ga03d4a1039b8662c71eeb40beea8cb622}{glfw\+Get\+Time}}.


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordtype}{double} seconds = \mbox{\hyperlink{group__input_ga03d4a1039b8662c71eeb40beea8cb622}{glfwGetTime}}();}

\end{DoxyCode}


It returns the number of seconds since the library was initialized with \mbox{\hyperlink{group__init_gab41771f0215a2e0afb4cf1cf98082d40}{glfw\+Init}}. The platform-\/specific time sources used typically have micro-\/ or nanosecond resolution.

You can modify the base time with \mbox{\hyperlink{group__input_ga94360a3628a09f32708f83cc3fa48590}{glfw\+Set\+Time}}.


\begin{DoxyCode}{0}
\DoxyCodeLine{\mbox{\hyperlink{group__input_ga94360a3628a09f32708f83cc3fa48590}{glfwSetTime}}(4.0);}

\end{DoxyCode}


This sets the time to the specified time, in seconds, and it continues to count from there.

You can also access the raw timer used to implement the functions above, with \mbox{\hyperlink{group__input_gaa00c3e32227eb70b3968fca0bfe4ae26}{glfw\+Get\+Timer\+Value}}.

\end{DoxyCode}


This value is in 1~/~frequency seconds. The frequency of the raw timer varies depending on the operating system and hardware. You can query the frequency, in Hz, with \mbox{\hyperlink{group__input_gaa92d10b10013372778efbf6367714371}{glfw\+Get\+Timer\+Frequency}}.
